## 一、"最强"的量化定义

**最强 Multi-Agent 协作平台 = 以最少人工介入，在最短时间内，完成最复杂的多步协作。**

四个可衡量的维度，以及每个维度的度量方法和当前基线：

| 维度 | 度量指标 | 目标 | 当前 v0.2 基线 |
|------|----------|:--:|:--:|
| **响应时效** | 事件从产生到 Agent 响应的 P50/P99 延迟 | P50 < 100ms, P99 < 500ms | 30s 轮询 (P50=15s) 🔴 |
| **自主比例** | Agent 自主完成步骤数 / 总步骤数 (不含人工) | > 90% | 0%（无 workflow 实际运行）🔴 |
| **协作深度** | 单次工作流中 Agent 间事件传递的最大链路长度 | > 5 步 | DAG 支持但未实测 |
| **鲁棒性** | 工作流失败后的自动恢复率 (含重试/回滚) | > 95% | 0%（无重试无回滚）🔴 |

**目标状态**：v0.7 结束时，响应时效 P50 < 1s（DAG 10s tick → sub-second event push），自主比例 > 80%。

## 三、AutoGen Orchestrator DAG 对标

### 3.1 AutoGen Orchestrator 架构

```
┌──────────────────────────────────────────┐
│         AutoGen Orchestrator             │
│                                          │
│  Task Queue (优先级队列)                   │
│      ↓                                   │
│  DAG Executor                            │
│      ├─ 依赖解析                          │
│      ├─ 条件分支 (输出值匹配)              │
│      └─ 失败回滚 (补偿任务)                │
│      ↓                                   │
│  Agent Pool (能力匹配 + 负载均衡)          │
│      ↓                                   │
│  死信队列 (DLQ + 重试)                    │
└──────────────────────────────────────────┘
```

### 3.2 条件分支 — AutoGen 怎么做

AutoGen 的条件分支不是"检查 step 是否完成"，而是 **检查 step 的输出值**：

```python
# AutoGen 条件分支（伪代码）
task_b = Task(
    condition=lambda ctx: ctx["task_a"].output["status"] == "approved",
    # 只有 task_a 输出 status=approved 时才执行
)

task_c = Task(
    condition=lambda ctx: ctx["task_a"].output["status"] == "rejected",
    # task_a 被拒绝时走另一条路
)
```

**vs Mind Agency v0.2**：
```typescript
// 我们只能检查 step 是否完成，不能检查输出值
condition: "step_a"           // step_a 完成了 → 执行
condition: "step_a.field"     // step_a 完成了 → 执行（field 未实际读取）
condition: "step_a.field == value"  // 只检查完成状态（Bug：未读取实际值）
```

**差距**：AutoGen 的条件分支是 **数据驱动** 的（基于输出值），我们是 **流程驱动** 的（基于完成状态）。在需要"审查通过 → 发布 / 审查驳回 → 返工"的场景中，我们的 condition 不够用。

### 3.3 失败回滚 — AutoGen 怎么做

AutoGen 通过 **补偿任务 (Compensation Task)** 实现回滚：

```python
# AutoGen 回滚（伪代码）
task_deploy = Task(
    action="deploy",
    compensation=Task(action="rollback_deploy")  # 失败时执行
)
```

执行流程：
```
deploy 成功 → 完成
deploy 失败 → 触发 compensation task (rollback_deploy) → 标记 deploy 为 FAILED_WITH_COMPENSATION
```

DLQ 捕获 compensation task 也失败的情况 → 人工介入。

**vs Mind Agency v0.2**：完全没有回滚概念。`StepStatus.FAILED` 枚举值已定义但从未被赋值——engine 只处理 PENDING → IN_PROGRESS → COMPLETED 的快乐路径。

### 3.4 对比总结

| 维度 | AutoGen Orchestrator | Mind Agency Scheduler v0.2 | 差距 |
|------|----------------------|---------------------------|:--:|
| **条件粒度** | 输出值匹配（ctx.task.output.field） | 完成状态检查 | 代际差距 |
| **分支路径** | 多路条件路由（if/elif/else） | 单条件（满足/跳过） | 代际差距 |
| **失败回滚** | 补偿任务（Compensation Task） | 无 | 缺失 |
| **重试** | DLQ + 指数退避 | 无（DLQ 在 EventBus 层，不在调度层） | 缺失 |
| **超时** | 有，Task 级别 | 字段已定义但未检查 | 轻微 |
| **并行** | gRPC 协程并发 | DAG 内串行 | 代际差距 |
| **优先级** | 优先级队列 | 无 | 缺失 |

## 四、CrewAI Flows 对标

### 4.1 CrewAI Flows 架构

```
┌───────────────────────────────────────┐
│           CrewAI Flows                │
│                                       │
│  @start()  入口                        │
│     ↓                                 │
│  @router() 条件路由 — 返回路由标签      │
│     ├─ "success" → @listen("success") │
│     └─ "failed"  → @listen("failed")  │
│     ↓                                 │
│  @listen() 事件驱动触发                │
│     ↓                                 │
│  or_() 任意触发 / and_() 全部触发      │
│     ↓                                 │
│  @human_feedback 暂停→人工→路由        │
└───────────────────────────────────────┘
```

### 4.2 条件分支 — CrewAI Flows 怎么做

```python
# CrewAI Flows 条件分支
@start()
def analyze():
    return "success" if result["score"] > 0.8 else "failed"

@listen("success")
def publish():
    ...

@listen("failed")
def retry():
    ...
```

**关键差异**：CrewAI 的条件分支是 **路由标签** 模式——方法返回一个字符串标签，后续方法按标签监听。这是一个隐式的有限状态机。

**vs Mind Agency v0.2**：
- CrewAI 的路由是 **返回值驱动**，我们是 **预先声明 condition 字符串**
- CrewAI 天然支持 N 路分支（任意标签），我们只有满足/跳过两路
- CrewAI 的 `and_()` 模式在我们的 `dependsOn` 已经有等价实现
- CrewAI 的 `or_()` 模式我们没有——只能通过多个 step 依赖同一个上游来模拟

### 4.3 失败处理 — CrewAI Flows 怎么做

CrewAI Flows 把失败处理交给了 **路由标签** :

```python
@start()
def process():
    try:
        result = do_work()
        return "success"
    except Exception as e:
        return "failed"

@listen("failed")
def handle_failure(result):
    # 重试或通知
    ...
```

**没有显式的补偿/回滚机制**——失败恢复靠开发者自己写 `@listen("failed")` 方法。回滚也靠手动实现。

### 4.4 对比总结

| 维度 | CrewAI Flows | Mind Agency Scheduler v0.2 | 差距 |
|------|-------------|---------------------------|:--:|
| **条件分支** | `@router()` 返回值 → N 路标签路由 | YAML condition 字符串（完成状态检查） | 代际差距 |
| **多触发** | `or_()` + `and_()` | `and_()` 等价于 dependsOn；`or_()` 缺失 | 缺失 |
| **人工介入** | `@human_feedback` 暂停 → 路由 | 无 | 缺失 |
| **失败回滚** | 无内置（靠路由标签手动实现） | 无 | 持平（都不好） |
| **事件驱动** | `@listen()` 方法级事件 | EventBus emit（系统级事件） | **我们领先** 🟢 |

## 七、结论

1. **条件分支是最大差距**：从"完成检查"升级到"输出值匹配"是调度器智能化的关键一步
2. **失败回滚 AutoGen 完胜**：补偿任务是业界标准做法，CrewAI 和我们都没有内置
3. **CrewAI Flows 的路由标签模式更轻量**：适合我们的轻量架构哲学，不学 AutoGen 的重量级补偿任务
4. **我们的事件集成优于 CrewAI**：CrewAI 的 `@listen()` 是方法级，我们的 EventBus 是系统级——所有前端 Dashboard 都能看到工作流进度
5. **建议 v0.3 兼顾两个来源**：条件分支学 CrewAI 的路由标签思路（轻量），失败回滚学 AutoGen 的补偿任务（但用 YAML 声明而非 Python 对象）
